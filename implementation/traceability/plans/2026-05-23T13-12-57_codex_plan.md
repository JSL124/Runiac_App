## Proposed Plan

1. **Executive recommendation**: Start Phase 1 with traceability and setup preparation, not production code. The first approved implementation task should be a traceability/setup document batch that maps PRD/PDD requirements to future Flutter, Firebase, Functions, rules, and tests.

2. **Phase 1 setup batches**:
   - Batch 0: verify repository state, protected paths, tool versions, target platforms, Firebase project ownership, and map provider choice.
   - Batch 1: create traceability files only after approval.
   - Batch 2: after separate approval, run `flutter create runiac_app` from `implementation/mobile/`.
   - Batch 3: after separate approval, run Firebase setup under `firebase/`.
   - Batch 4: prepare emulator and test harness boundaries before trusted backend logic.
   - Batch 5: only then begin minimal app/backend integration.

3. **Before `flutter create`**: confirm `implementation/mobile/runiac_app/` does not exist, package/app ID, iOS/Android targets, Flutter SDK version, Firebase Flutter plugin approach, map provider, location/privacy permission wording, and that protected PDD/submission paths remain untouched.

4. **Before `firebase init`**: confirm Firebase project, environment naming, region, emulator targets, Functions runtime, Firestore rules testing approach, FCM needs, optional Storage scope, billing implications, and that no production data/rules deployment will occur.

5. **Folder/file creation order**: traceability docs first, Flutter scaffold second, Firebase scaffold third, emulator/test harness fourth, shared contracts/models fifth, production feature code last.

6. **Backend ownership safeguards**: Cloud Functions must own validation, XP, streak, level, rank, weekly XP, monthly XP, leaderboard score, aggregation, entitlement-sensitive generation, and expert-plan governance. Flutter may submit raw/user-owned inputs and display trusted backend results only.

7. **Flutter/Firebase boundary plan**: Flutter owns UI, navigation, GPS tracking UI, local interaction, temporary run state, and client integration. Firebase Auth owns identity. Firestore stores app data. Cloud Functions own trusted mutations and derived values. FCM handles notifications. Storage remains optional media storage only.

8. **Traceability plan**: create requirement maps from PRD features F1-F10 and submitted/internal PDD design elements to implementation tasks, affected future files, security rules, tests, and acceptance evidence. MVP priority should follow PRD: F1, F2, F3, F4, F6, F9 before Phase 2 items F5, F7, F8, F10.

9. **Safe representation**: `subscriptionStatus` represents Basic/Premium access only. `userRole` represents Platform Administrator and Medical Trainer/Expert operational/governance access. Neither field should be user-writable from Flutter.

10. **Testing/emulator preparation**: prepare Firebase Emulator Suite strategy before production logic: Auth, Firestore, Functions, rules tests, seed data, role/entitlement fixtures, XP tampering attempts, Medical Trainer/Expert publish-denial checks, admin-only governance checks, and Flutter integration plans.

11. **Do not implement in Phase 1**: no XP formulas, leaderboard aggregation, AI/LLM summaries, production Firestore rules, Storage rules, GitHub Actions, real deployments, expert dashboards, admin dashboards, full map tracking, or premium competitive advantages.

12. **Recommended first approved implementation task**: create Phase 1 traceability/setup files under `implementation/traceability/` only, documenting requirement-to-task mapping, approval gates, backend ownership rules, and emulator strategy.
