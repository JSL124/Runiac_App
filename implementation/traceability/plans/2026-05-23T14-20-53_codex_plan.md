## Proposed Plan

1. Executive recommendation

Implement the efficiency features after the generic profile migration, or combine them only after the profile migration scope is approved. These review settings belong partly in runner logic and partly in Runiac profile prompts/config, so adding them before migration would likely create duplicate churn in legacy prompt paths.

If profile migration is deferred, the efficiency batch can still be implemented against the current `tools/agent-review/` layout, but it should be kept small and avoid moving files.

2. Proposed implementation sequence

1. Finish or explicitly defer generic profile migration.
2. Add review configuration parsing with safe defaults.
3. Add short review packet generation for lite review.
4. Add provider routing without silent fallback.
5. Add manual checklist mode.
6. Update README/config examples.
7. Run only approved dry-run smoke checks after implementation approval.

3. Minimal first batch

Smallest safe first batch:

- Add `REVIEW_MODE`, `REVIEW_PROVIDER`, and `DECISION_POLICY` config defaults.
- Keep current behavior as default: `standard`, `claude`, `manual`.
- Add `manual`/`none` reviewer path that produces instructions/checklist instead of calling Claude.
- Add explicit Codex fallback only when the user configures `REVIEW_PROVIDER=codex`.
- Do not yet optimize every prompt or introduce complex packet caching.

4. Config variables and defaults

Recommended defaults:

```bash
REVIEW_MODE="standard"          # lite | standard | full
REVIEW_PROVIDER="claude"        # claude | codex | none
DECISION_POLICY="manual"        # always | if-needed | manual
```

Meaning:

- `REVIEW_MODE=lite`: short packet review only.
- `REVIEW_MODE=standard`: current plan review behavior.
- `REVIEW_MODE=full`: broader review packet, more context, slower.
- `REVIEW_PROVIDER=claude`: use Claude only; fail clearly if unavailable.
- `REVIEW_PROVIDER=codex`: explicitly use Codex as reviewer.
- `REVIEW_PROVIDER=none`: skip automated reviewer and emit manual checklist.
- `DECISION_POLICY=manual`: current safest behavior; user inspects decision before implementation.
- `DECISION_POLICY=if-needed`: only later, after evidence from real runs.
- `DECISION_POLICY=always`: should still never run implementation.

5. Review packet design

Claude-lite should review a short packet containing:

- task prompt
- generated plan summary
- changed-file scope from plan
- untouched-file scope
- Runiac non-negotiable constraints
- git status summary
- approval gate
- explicit question list for reviewer

Avoid sending full PRD/PDD text unless `REVIEW_MODE=full`.

6. Claude-lite strategy

Use Claude with the existing safe flags:

```bash
claude --permission-mode plan --tools "Read,Grep,Glob"
```

Lite mode should ask Claude for concise risk review only, not implementation patches. It must not use Bash/Edit/Write or unsafe permission modes.

7. Codex fallback strategy

Do not silently fall back from Claude to Codex.

If `REVIEW_PROVIDER=claude` and Claude is missing, fail with a clear message and suggest rerunning with:

```bash
REVIEW_PROVIDER=codex
```

Only use Codex review when the user explicitly selects it.

8. Manual checklist strategy

For `REVIEW_PROVIDER=none`, generate a manual review checklist artifact covering:

- unsafe file operations
- PDD/submitted baseline drift
- Flutter/Firebase boundary drift
- XP/leaderboard/backend ownership
- subscription/userRole separation
- approval gate before implementation
- no git staging/commit/push

9. Risks and acceptance checks

Main risks:

- Silent fallback could hide provider failures.
- Lite packet may omit context needed for safety review.
- `DECISION_POLICY=always` could be misunderstood as implementation approval.
- Profile migration overlap could cause duplicated edits.
- Manual mode could be mistaken for passed automated review.

Acceptance checks:

- Pipeline remains only `plan -> review -> decision`.
- Implementation remains separate and user-approved.
- No PRD/PDD/submitted PDF/Flutter/Firebase/tests changes.
- No `danger-full-access`.
- No `workspace-write` with approval `never`.
- Missing Claude fails clearly unless `REVIEW_PROVIDER=codex` is explicitly set.
- Premium still receives no XP/rank/leaderboard advantage.
- Cloud Functions remain owner of trusted XP, streak, level, rank, validation, aggregation, entitlement, and expert-plan governance logic.
