## Proposed Plan

1. Prepare `requirements-map.md` as a traceability matrix, not an implementation spec.

2. Use this proposed structure for `requirements-map.md`:
   - Document purpose and Phase 1 scope.
   - Source hierarchy and reference rules.
   - Runiac invariants summary.
   - Requirements map table with columns:
     - Feature area
     - PRD/PDD source reference
     - Phase 1 priority
     - Implementation owner area
     - Backend-owned fields
     - Security/access-control concerns
     - Scaffold dependency
     - Minimally Viable Test Criterion / Test Assertions
   - Deferred / out-of-scope requirements section.
   - Open questions and approval dependencies.
   - Review checklist before implementation starts.

3. Suggested `requirements-map.md` feature areas:
   - Authentication and user identity
   - User profile and sensitive running data
   - Basic/Premium access control
   - Running activity capture and GPS route handling
   - Plans and expert-plan governance
   - XP, streak, level, rank, and leaderboard
   - Firestore data ownership
   - Cloud Functions trusted logic
   - Notifications or messaging, if Phase 1 includes them
   - AI/LLM boundaries, if any future feature proposes AI assistance

4. Prepare `setup-gates.md` as an approval checklist that must be completed before production scaffolding.

5. Use this proposed structure for `setup-gates.md`:
   - Document purpose and non-scaffolding warning.
   - Gate status legend: `Not started`, `Blocked`, `Ready for review`, `Approved`, `Deferred`.
   - Approval table with columns:
     - Gate
     - Required evidence
     - Owner/reviewer role
     - Approval criteria
     - Blocking risks
     - Status
     - Notes / decision link
   - Gate-by-gate checklist.
   - Final “Ready for Phase 1 scaffold” decision section.

6. Required `setup-gates.md` gates:
   - Flutter scaffold approval
   - Firebase config approval
   - Firestore data model approval
   - Cloud Functions boundary approval
   - Security Rules approval
   - Secret/API key/env handling approval
   - `.gitignore` checks for generated config files
   - GPS/privacy approval
   - XP/leaderboard backend ownership approval
   - Basic/Premium access-control approval
   - App Check decision
   - AI/LLM boundary approval

7. Apply always-on Runiac invariants in both documents:
   - Flutter owns UI, navigation, GPS tracking UI, local interaction, and client integration.
   - Firebase Authentication owns identity.
   - Firestore stores app data.
   - Cloud Functions own trusted XP, streak, level, rank, leaderboard, validation, aggregation, entitlement, and expert-plan governance logic.
   - Flutter may display trusted XP/rank/leaderboard values but must not directly write them.
   - `subscriptionStatus` controls Basic/Premium access.
   - `userRole` controls operational/governance access.
   - Medical Trainer/Expert submits draft expert plans only.
   - Platform Administrator approves, rejects, publishes, and archives expert plans.
   - Premium must not create competitive advantage.
   - AI/LLM must not become official XP/rank/leaderboard logic.

8. Claude scope-limited review should verify:
   - The two proposed documents stay traceability-only.
   - The matrix captures test assertions before implementation.
   - Setup gates block scaffolding until ownership, security, privacy, secrets, and access-control decisions are approved.
   - No proposed step modifies PRD/PDD, diagrams, wireframes, submitted docs, production source, tests, or config.

9. Codex final decision after review:
   - Approve only if the plan remains limited to the two traceability files.
   - Defer if review requires more than the six-file review scope.
   - Block if any step implies Flutter/Firebase scaffolding, production config creation, secret handling, or PRD/PDD rewriting.
