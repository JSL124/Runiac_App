## Missing Constraints

- **CLAUDE.md system prompt must be preserved across all modes.** The current review step passes `--append-system-prompt "$(cat CLAUDE.md)"`, which is what carries the read-only guardrails into Claude. The plan's packet/lite design does not state this must be retained. Add an explicit constraint that lite/standard/full and any provider path keep the guardrail system prompt and the `Read,Grep,Glob`-only tool restriction.
- **Checklist/packet output location and overwrite safety.** The plan does not say where the `none`-mode checklist is written or that it must go through `ensure_new_file`. Constrain it to not overwrite and not masquerade as a Claude review.
- **When lite is and is not allowed.** No rule prevents lite mode on a plan that touches XP/leaderboard/Firebase/security boundaries — exactly where full context matters most.
- **Provider precedence vs config/profile.** With the profile migration also in flight, the interaction between `REVIEW_PROVIDER`/`REVIEW_MODE`/`DECISION_POLICY` and `AGENT_REVIEW_CONFIG`/`AGENT_REVIEW_PROFILE` precedence is undefined. The prior decision required deterministic precedence; this plan should inherit that.
- **README content obligations.** README is listed as affected, but the plan does not require documenting that `REVIEW_PROVIDER=codex` disables independent review and that `none`/manual mode is not a passed review.

## Unsafe Operations

- No destructive operations are proposed. The plan preserves `DRY_RUN=1`, the read-only sandbox for plan/decision, `workspace-write`+`on-request` for implementation, no `danger-full-access`, no `workspace-write` with `--ask-for-approval never`, and explicit per-file `git add`. This is compliant.
- One latent issue: `REVIEW_PROVIDER=none` makes the runner *write a file even when no agent runs*. That write path must honor the existing dry-run preview behavior and `ensure_new_file` guard so it cannot overwrite prior artifacts or create directories in dry-run.
- The plan tells the user not to stage `…14-20-53_codex_plan.md`, but that file is the traceability artifact of *this very task*, not an unrelated change. Calling it "unrelated" is slightly misleading; the real intent (don't bundle traceability artifacts into a code commit) is fine, but the wording should be corrected.

## PRD/PDD Consistency Issues

- No direct PRD or submitted-PDD content is touched — this is dev tooling, and the plan keeps `PRD.md`, `docs/submissions/pdd/`, and the submitted PDF untouched. Good.
- Indirect process risk: the reviewer's charter (per `02_claude_review_plan.md` and CLAUDE.md) explicitly includes catching PRD/submitted-PDD/current-PDD mismatch. The lite-packet "avoid sending PDD text" optimization could erode that safeguard. Because the submitted PDD is a frozen PDF (not readable as text by the packet assembler anyway), the reviewer relies on its `Read` tool access to `docs/submissions/pdd/` — so any mode that trims tool access or context would directly weaken the baseline-drift check. Keep PDD reachability intact in every mode.

## Overengineering Concerns

- **27-combination config matrix for a single in-repo consumer.** `REVIEW_MODE` (lite/standard/full) × `REVIEW_PROVIDER` (claude/codex/none) × `DECISION_POLICY` (always/if-needed/manual) is far beyond current need. The README itself says to wait for 3–5 real tasks before even externalizing; only ~2–3 have run.
- **Defining values that are explicitly "for later."** `DECISION_POLICY=if-needed`/`always` and arguably `REVIEW_MODE=full` are documented but not meant to be wired up now. Defining unused enum values is dead config that invites misuse — drop them until justified.
- **The "review packet" generator is a new subsystem.** Today the runner simply concatenates prompt + plan and lets Claude read the repo via tools. Building a packet assembler (plan summary, changed/untouched scope extraction, git summary, question list) is meaningful new complexity for marginal token savings on a workflow run a handful of times. `REVIEW_MODE=full` "more context" is largely redundant with the reviewer's existing full tool-driven repo access.
- The plan's **own "minimal first batch" is the right scope**: defaults + `manual`/`none` path + explicit (non-silent) `codex` opt-in, deferring packet caching and prompt optimization. The recommendation below is essentially to ship that and discard the rest until evidence accrues.

## Recommendations

1. **Resolve sequencing first.** Decide migrate-first vs minimal-batch-first as a hard gate before any implementation, since both this plan and decision `13-56-50` edit `run_plan_review.sh`, `README.md`, and the prompts. Do not run both in parallel.
2. **Ship only the minimal batch.** Implement `REVIEW_MODE` (default `standard`), `REVIEW_PROVIDER` (default `claude`), and the `none`/manual-checklist path. Defer `lite`/`full`, the packet generator, `DECISION_POLICY=if-needed`/`always`, and prompt-token optimization until ≥3 real runs justify them.
3. **Treat `REVIEW_PROVIDER=codex` as high-risk.** Prefer omitting it now. If included, it must fail-loud with an explicit "independent review disabled" warning, never be default, and be documented in the README "Safety" section.
4. **Keep Runiac rules out of the runner.** Generate the manual checklist and any packet content from `prompts/` (or `profiles/runiac/`), not from `run_plan_review.sh`/`common.sh`, to respect the documented layering.
5. **Make `none`-mode unmistakably "not a review."** Distinct filename, in-file label, `ensure_new_file` protection, and dry-run-aware output.
6. **Preserve guardrails in every mode.** Keep `--append-system-prompt "$(cat CLAUDE.md)"`, `--permission-mode plan`, and `--tools "Read,Grep,Glob"`; default all new vars with `${VAR:-default}` after sourcing config to stay safe under `set -u`.
7. **Fix the commit metadata.** Use `chore(agent-review): ...` to match history, and reword the "unrelated file" note so the task's own traceability plan isn't mislabeled.
8. **Keep the approval gate and manual commits exactly as proposed** — this part is correct and should not change.
